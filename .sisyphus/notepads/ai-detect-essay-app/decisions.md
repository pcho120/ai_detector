## Task 2 Fix

## Task 3 â€” .docx Extraction & Quality Checks

- `extractDocx` accepts a `Buffer` directly (not a file path) to keep the extraction API decoupled from temp-file management. Callers (task 7 orchestration) combine it with `withTempFile`.
- Quality check order: (1) parse/throw on error, (2) check empty after normalize, (3) check garbled, (4) check min length, (5) check max length. This order ensures each failure returns the most informative error code.
- Garbled-text detection uses two independent heuristics (alphanumeric ratio and control-char ratio) so that either alone is sufficient to reject. False-positive risk on valid essays is negligible because even non-English essays have high alphanumeric ratios.
- Mammoth `messages` with type `'error'` are treated as a hard failure (`EXTRACTION_FAILED`), not merely surfaced as warnings. This ensures unusual but technically-parsed documents (e.g., encrypted content layers that mammoth partially reads) still fail.
- Hand-written `src/types/mammoth.d.ts` covers only the APIs used (`extractRawText`, `convertToHtml`, `MammothResult`, `MammothMessage`, `MammothOptions`). No attempt to type the full mammoth API surface.

## Task 5 â€” Normalized Detection Adapter (2026-03-30)

- `DetectionAdapter` interface defined in `src/lib/detection/types.ts` as the sole public contract for callers (Task 7, Task 9). All UI and route code will depend only on this interface, never on `SaplingDetectionAdapter` directly.
- `SaplingDetectionAdapter` is a class (not a function) to allow future subclassing or testing via constructor injection of alternate `fetch` implementations if needed, while remaining compatible with the `DetectionAdapter` interface.
- Provider error response body (`msg` field) is intentionally NOT included in the `DETECTION_FAILED` message. HTTP status code is the only safe diagnostic surfaced to callers. This was validated by a test that checked the error message did not include a simulated postgres connection string.
- `normalizeSaplingResponse` exported as a standalone pure function (not a private method) to enable direct unit testing of the normalization path without needing to mock `fetch`.
- Score convention: normalized score = Sapling native score, no transformation. `0` = human-like, `1` = AI-like, matching plan spec exactly.
- Request timeout is 30 seconds via `AbortController`. Timeout errors surface as `DETECTION_FAILED` with a "timed out" message distinct from network errors.

## Task 6 â€” Sentence Matching & Highlight Span Generation (2026-03-30)

- `SentenceScore` interface is defined locally in `src/lib/highlights/spans.ts` rather than importing from `src/lib/detection/`. This avoids a circular dependency and keeps the highlights utility self-contained. Task 7 will cast the detection adapter output to `SentenceScore[]` (they are structurally compatible).
- Two threshold constants (`HIGH_RISK_THRESHOLD = 0.7`, `MEDIUM_RISK_THRESHOLD = 0.4`) are exported from `spans.ts` so task 8 UI can reference them for consistent bucket rendering without re-encoding the same magic numbers.
- The sliding-window algorithm is bounded at `needleLen+20` chars rather than `needleLen*2+20` from the initial design. Essay sentences are typically 60â€“200 chars; a `*2` multiplier would be 120â€“400 chars, creating quadratic O(textLen * needleLen) scan at each position. The `+20` flat tolerance is sufficient for surrounding punctuation and whitespace edge cases while keeping the inner loop short.
- Anchor character skip includes `.,;:!?` in addition to `\s`. This is required to prevent the scanner anchoring on a trailing sentence-terminal char from the previous sentence and producing a loose-normalisation match that consumes the inter-sentence gap.
- `buildHighlightSpans` is the only exported function that modifies state (the `lastMatchEnd` map); it is a pure transform with no side effects on its inputs.

## Task 7 â€” Analysis Route & Orchestration Pipeline (2026-03-30)

- `SuggestionService` wired as a stable seam returning `[]` (noop) until task 9 owns real implementation. `NoopSuggestionService` lives in `src/lib/suggestions/noop.ts`.
- Route uses `export const runtime = 'nodejs'` to ensure Node runtime for Buffer, mammoth, and word-extractor compatibility.
- `isEnglish()` uses a two-pass letter ratio: total letters (`[a-zA-Z\u00C0-\u024F]`) vs basic Latin (`[a-zA-Z]`). If basic Latin / total letters < 0.6, the text is flagged as non-English. This correctly rejects Cyrillic and other non-Latin scripts.
- `isGarbled()` in docx.ts updated to use Unicode property escapes `\p{L}\p{N}` (with `u` flag) instead of ASCII-only `[a-zA-Z0-9]`. This allows Cyrillic/non-Latin DOCX files to pass extraction so the language check can catch them with `UNSUPPORTED_LANGUAGE` (rather than `EXTRACTION_FAILED`).
- Detection adapter creation is separated from its usage so a missing API key returns 503 (service unavailable) while a detection call failure returns 502 (bad gateway). The status codes semantically distinguish config errors from runtime failures.
- All error responses go through `toErrorResponse(err)` which strips to `{error, message}` only â€” no raw paths, no provider internals.

## Task 8 â€” UI Upload & Review Panel

- **API Modification**: Extended the `/api/analyze` success response to return the raw `text`. This was strictly necessary because the `highlights` (which contain `start` and `end` indices) point to absolute indices in the originally extracted text (including raw spacing and paragraphs) â€” relying on concatenated `sentences` array would lead to drift and incorrect highlighting.
- **Span Rendering Strategy**: Used standard JSX array accumulation by keeping track of a `currentIndex` variable and pushing alternating un-highlighted text (`text.slice(currentIndex, highlight.start)`) and highlighted `<span>` elements (`text.slice(highlight.start, highlight.end)`). Avoided `dangerouslySetInnerHTML`.
- **E2E Testing**: Used Playwright's `page.route` to mock the `/api/analyze` response for `home.spec.ts`, rather than executing the real API logic. This keeps tests deterministic, decoupled from test fixtures in the UI tests, and fast.

## Task 9 â€” Safe Suggestion Generation Service (2026-03-30)

- **No external LLM**: `RuleBasedSuggestionService` uses 12 regex-backed coaching rules instead of an LLM call. This eliminates any additional API key dependency, keeps the service self-contained, and ensures server-side-only execution with zero latency overhead beyond regex matching.
- **Guardrails as separate module**: `src/lib/suggestions/guardrails.ts` exports `containsBannedPhrase` and `applyGuardrails` as standalone pure functions. This separation makes the safety gate independently testable without instantiating the service.
- **One suggestion per sentence**: `RuleBasedSuggestionService.suggest()` breaks after the first matching rule for each sentence. This avoids overwhelming output and keeps suggestions atomic/focused.
- **Coaching direction, not full rewrite**: Each rule's `rewriteHint` is a directional coaching prompt (20â€“150 chars) that tells the writer what to do, not a substitute sentence. This honours the "no full-essay rewrites" constraint.
- **Route wiring**: `createSuggestionService()` in `route.ts` now returns `RuleBasedSuggestionService` instead of `NoopSuggestionService`. The `NoopSuggestionService` is preserved in `noop.ts` for testing fallback scenarios.
- **Integration test update**: Replaced the `'returns suggestions as empty array (noop service)'` assertion with a shape-validation test. Added two new integration tests: one asserting suggestions are populated when high-risk sentences match patterns (using `sapling-ai-like.json` mock), and one asserting no evasion language in output.

- Documented Vercel Node.js runtime requirement and SAPLING_API_KEY env variable in README.md.
- Opted for rule-based suggestion service over LLM to maintain zero-latency and zero-extra-cost baseline for v1.
